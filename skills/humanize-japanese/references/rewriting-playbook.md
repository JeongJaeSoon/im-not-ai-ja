# 日本語リライト・プレイブック

## 基本手順

1. 原文の目的、読者、媒体、レジスターを一行で記録する。
2. Do-NOT span を固定する。人名・製品名・組織名などは明示的な保護語として記録する。
3. taxonomy の ID が付く箇所だけ編集する。
4. 定型句 → 構成 → 接続 → 統語・語彙 → リズム → 装飾の順に直す。
5. 原文と文単位で照合し、モダリティと待遇関係を監査する。

## 置換ではなく意味を選ぶ

| Before | 危険な機械置換 | 判断例 |
|---|---|---|
| `これにより効率が上がる` | 常に `その結果` | 因果が明白なら `処理が速くなる` と直接つなぐ |
| `導入することができます` | 常に `導入する` | 能力なら `導入できる`、許可なら `導入してよい` |
| `重要です` | 常に削除 | 規範性が論旨なら理由を残して簡潔にする |
| `期待されます` | 常に断定 | 期待主体がない場合のみ削る。予測を断定へ変えない |
| `させていただきます` | 常に `します` | 許可・受益があるかを register guide で確認 |

## ジャンルガード

| ジャンル | 残すもの | 注意点 |
|---|---|---|
| 学術 | `である`、受動、名詞化、定義、引用 | 根拠のない誇張だけ外す。用語を口語化しない |
| 仕様・技術文書 | 箇条書き、定型見出し、短文 | 均一さやコロンを AI tell にしない |
| 法務・規程 | 義務、例外、定義、定型句 | 原則リライトせず指摘だけ返す |
| プレスリリース | 丁寧体、社名、定型の問い合わせ | 宣伝語の削除で公式見解を弱めない |
| 社外メール | R3、依頼・謝罪・通知の定型 | 行為者と敬語の向きを確認 |
| 友人チャット | R0、句読点の省略、俗語 | 丁寧体へ正規化しない |
| やさしい日本語 | 短文、語彙制限、明示的主語 | 文長の均一性を欠点にしない |
| 小説・随筆 | 比喩、反復、ダッシュ、揺れ | 計量閾値を使わず作者の声を優先 |

## 変更率

文字単位の差分率を目安にする。

- 0〜10%: 軽微な編集。
- 10〜25%: 通常の範囲。
- 25〜30%: 意味監査を強化。
- 30〜50%: 警告し、編集単位ごとに根拠を要求。
- 50%超: 中止して最後の安全版へ戻す。

ただし100字未満または5文未満では、文字単位の変更率を合否判定に使わない。短い一文は一箇所の修正でも比率が大きくなるため、変更箇所数、保護語、モダリティ、レジスターを直接監査する。変更不要の0%は成功である。

比較時は必要に応じて保護語を明示する。

```bash
scripts/metrics.py before.md --compare after.md \
  --protect "佐藤美咲" --protect "Project KAI"
```

## 禁止事項

- 人間らしく見せるための誤字、俗語、個人的体験の捏造。
- AI 検出器のスコアを目的にした無意味な言い換え。
- 丁寧体を「機械的」という理由だけで崩すこと。
- `かもしれない`、`必要がある`、`禁止する` などの強度変更。
- 数値、引用、法令、製品名、コードの整形を含む変更。
